Dominando las Subconsultas
en MySQL

Una subconsulta es un SELECT dentro de otro SELECT. Te permite usar el resultado de una consulta como condición, tabla o valor en otra — sin necesidad de guardar el resultado intermedio.

¿Qué es una subconsulta?

Una subconsulta (también llamada subquery o consulta anidada) es una sentencia SELECT que vive dentro de otra sentencia SQL. La consulta interior se ejecuta primero y su resultado es usado por la consulta exterior.

🪆
Analogía de las muñecas rusas Una subconsulta es como una matryoshka: hay una muñeca dentro de otra. Cuando abres la muñeca exterior (consulta principal), primero tienes que resolver qué hay adentro (subconsulta). El resultado de la muñeca interior se convierte en parte de la exterior. Podés tener varios niveles de anidación.
Consulta exterior
La consulta principal. Usa el resultado de la subconsulta como condición, tabla o valor calculado.
( )
Paréntesis obligatorios. La subconsulta SIEMPRE va entre paréntesis. Sin ellos MySQL da error de sintaxis.
Consulta interior
La subconsulta. Se ejecuta primero. Su resultado alimenta a la consulta exterior.
Alias (AS)
Obligatorio en FROM. Cuando la subconsulta va en el FROM, siempre necesita un alias.
SELECT nombre FROM empleados WHERE salario = ( -- Subconsulta: se ejecuta primero SELECT MAX(salario) FROM empleados );

Tipos de subconsulta según su ubicación

Una subconsulta puede ir en distintas partes de la consulta principal, y su comportamiento cambia según dónde esté.

En WHERE
Filtra filas usando el resultado como condición. El más común.
En FROM
La subconsulta actúa como una tabla temporal. Necesita alias.
En SELECT
Calcula un valor por cada fila del resultado.
En HAVING
Filtra grupos usando el resultado como condición.

Subconsulta en WHERE

Es el uso más frecuente. La subconsulta calcula un valor o lista de valores, y el WHERE filtra las filas que cumplen esa condición.

🔍
Analogía del filtro dinámico Imagina que querés los empleados que ganan más que el promedio, pero no sabés el promedio de memoria. La subconsulta calcula ese promedio automáticamente y lo usa como filtro. Es un filtro que se recalcula solo cada vez.
Empleados que ganan más que el promedio
SELECT nombre, salario FROM empleados WHERE salario > ( SELECT AVG(salario) FROM empleados ); -- La subconsulta devuelve: 2421.67 -- WHERE filtra salarios mayores a ese valor
La subconsulta calcula AVG = 2421.67 → Resultado:
nombresalario
Ana Torres3410
Carmen Ríos3520
El empleado con el salario más alto
SELECT nombre, salario FROM empleados WHERE salario = ( SELECT MAX(salario) FROM empleados );
Resultado →
nombresalario
Carmen Ríos3520
📌
Subconsulta escalar vs de lista Si la subconsulta devuelve un solo valor (como MAX o AVG), usa =, >, <. Si devuelve múltiples valores, usa IN, NOT IN, ANY, ALL.

Subconsulta con IN y NOT IN

Cuando la subconsulta devuelve múltiples valores (varias filas), usás IN para verificar si un valor está en esa lista, o NOT IN para lo contrario.

Empleados que tienen al menos un pedido
SELECT nombre, ciudad FROM empleados WHERE id IN ( SELECT DISTINCT empleado_id FROM pedidos ); -- Subconsulta devuelve: (1, 2, 3, 5) -- WHERE filtra los empleados con esos ids
La subconsulta devuelve ids: 1, 2, 3, 5 → Resultado:
nombreciudad
Ana TorresQuito
Luis MoraGuayaquil
Carmen RíosQuito
Sofía LeónGuayaquil
Empleados que NO tienen ningún pedido
SELECT nombre, ciudad FROM empleados WHERE id NOT IN ( SELECT DISTINCT empleado_id FROM pedidos );
Resultado → Pedro Vega y Marco Díaz no tienen pedidos
nombreciudad
Pedro VegaCuenca
Marco DíazQuito
⚠️
NOT IN con NULLs es peligroso Si la subconsulta devuelve algún NULL, NOT IN no devuelve ninguna fila. Esto es un bug silencioso muy común. Preferí NOT EXISTS cuando pueda haber NULLs.

Subconsulta con EXISTS y NOT EXISTS

EXISTS verifica si la subconsulta devuelve al menos una fila. No le importa qué valor devuelve — solo si existe o no. Es más eficiente que IN para tablas grandes.

🚦
Analogía del semáforo EXISTS es como un detector de presencia: solo necesita saber si hay algo, no qué hay. Si detecta al menos una fila, devuelve verdadero (verde). Si no detecta nada, devuelve falso (rojo). La consulta exterior actúa según ese semáforo.
Empleados que tienen al menos un pedido (EXISTS)
SELECT e.nombre, e.ciudad FROM empleados e WHERE EXISTS ( SELECT 1 FROM pedidos p WHERE p.empleado_id = e.id ); -- "SELECT 1" es convención: EXISTS solo verifica -- si hay filas, no qué columna se devuelve
Empleados SIN pedidos (NOT EXISTS)
SELECT e.nombre, e.ciudad FROM empleados e WHERE NOT EXISTS ( SELECT 1 FROM pedidos p WHERE p.empleado_id = e.id );
EXISTS
-- Más eficiente que IN -- para tablas grandes. -- Se detiene al encontrar -- la primera coincidencia. -- Seguro con NULLs.
IN
-- Más simple de leer. -- Puede ser lento si la -- subconsulta devuelve -- miles de valores. -- Problemático con NULLs.

Subconsulta en FROM — tabla derivada

Cuando la subconsulta va en el FROM, se convierte en una tabla temporal (también llamada tabla derivada). La consulta exterior consulta esa tabla como si fuera real. Siempre necesita un alias.

🏗️
Analogía del andamio Una tabla derivada es como un andamio de construcción: no es parte del edificio final, pero te permite construirlo. Calculás datos intermedios en la subconsulta (el andamio), y la consulta exterior los usa para obtener el resultado final.
Departamentos con salario promedio mayor a 2000
SELECT dept_stats.departamento, dept_stats.promedio FROM ( SELECT departamento, ROUND(AVG(salario), 2) AS promedio FROM empleados GROUP BY departamento ) AS dept_stats WHERE dept_stats.promedio > 2000;
La subconsulta calcula promedios por departamento → Resultado:
departamentopromedio
Ingeniería3465.00
Ventas2000.00
💡
¿Por qué usar subconsulta en FROM? Porque no podés usar alias de funciones de agregado directamente en WHERE. Por ejemplo, WHERE AVG(salario) > 2000 da error. La subconsulta en FROM resuelve esto al calcular el promedio primero como una "tabla" separada.

Subconsulta en SELECT — valor calculado

Cuando la subconsulta va dentro del SELECT, calcula un valor por cada fila del resultado principal. Debe devolver exactamente un valor (escalar).

Cada empleado con el salario máximo de su departamento
SELECT nombre, departamento, salario, ( SELECT MAX(salario) FROM empleados e2 WHERE e2.departamento = e1.departamento ) AS max_dpto FROM empleados e1;
Resultado → cada fila muestra el máximo de su propio departamento

Subconsulta correlacionada El ejemplo anterior es una subconsulta correlacionada: la subconsulta interior referencia una columna de la consulta exterior (e1.departamento). Se ejecuta una vez por cada fila de la consulta exterior — puede ser lenta en tablas grandes.

Subconsulta en HAVING

Podés usar subconsultas en HAVING para comparar estadísticas de grupos contra un valor calculado dinámicamente.

Departamentos cuyo salario promedio supera el promedio global
SELECT departamento, ROUND(AVG(salario), 2) AS promedio_dpto FROM empleados GROUP BY departamento HAVING AVG(salario) > ( SELECT AVG(salario) FROM empleados ); -- Subconsulta calcula el promedio global: 2421.67 -- HAVING filtra departamentos que lo superan
Resultado →
departamentopromedio_dpto
Ingeniería3465.00

Subconsulta vs JOIN — ¿cuándo usar cada uno?

La misma consulta muchas veces se puede escribir con subconsulta o con JOIN. Cada enfoque tiene sus ventajas.

Con subconsulta
SELECT nombre FROM empleados WHERE id IN ( SELECT empleado_id FROM pedidos );
Con JOIN equivalente
SELECT DISTINCT e.nombre FROM empleados e INNER JOIN pedidos p ON e.id = p.empleado_id;
SubconsultaJOIN
LegibilidadMás clara para lógica complejaMás clara para combinar columnas
RendimientoPuede ser lenta (correlacionada)Generalmente más rápido con índices
Columnas extraSolo filtra, no agrega columnasPuede traer columnas de ambas tablas
Valores calculadosIdeal (MAX, AVG como filtro)Más complejo para esto
NOT EXISTSNatural y seguroNecesita LEFT JOIN + IS NULL
📌
Regla práctica Usá subconsulta cuando necesitás comparar contra un valor calculado (MAX, AVG, COUNT) o verificar existencia (EXISTS). Usá JOIN cuando necesitás columnas de ambas tablas en el resultado o cuando el rendimiento es crítico.

Errores comunes

1. Subconsulta sin paréntesis

✗ Error de sintaxis
SELECT nombre FROM empleados WHERE salario > SELECT AVG(salario) FROM empleados; -- Falta el paréntesis
✓ Correcto
SELECT nombre FROM empleados WHERE salario > ( SELECT AVG(salario) FROM empleados ); -- Siempre entre ( ) ✓

2. Subconsulta en FROM sin alias

✗ Error — falta alias
SELECT departamento FROM ( SELECT departamento, AVG(salario) AS prom FROM empleados GROUP BY departamento ); -- Tabla derivada necesita -- un alias obligatorio
✓ Con alias
SELECT stats.departamento FROM ( SELECT departamento, AVG(salario) AS prom FROM empleados GROUP BY departamento ) AS stats; -- "AS stats" es el alias ✓

3. Usar = cuando la subconsulta devuelve múltiples filas

✗ Error en ejecución
SELECT nombre FROM empleados WHERE id = ( SELECT empleado_id FROM pedidos ); -- Devuelve 5 filas, no 1 -- "=" espera un solo valor
✓ Usar IN
SELECT nombre FROM empleados WHERE id IN ( SELECT empleado_id FROM pedidos ); -- IN acepta múltiples -- valores ✓

Resumen: subconsultas completo

Ejemplo completo — reporte avanzado
-- Empleados activos que ganan más que el promedio -- de su departamento, con el promedio global como referencia SELECT e.nombre, e.departamento, e.salario, dept.promedio_dpto, (SELECT ROUND(AVG(salario),2) FROM empleados) AS promedio_global FROM empleados e INNER JOIN ( SELECT departamento, ROUND(AVG(salario),2) AS promedio_dpto FROM empleados GROUP BY departamento ) AS dept ON e.departamento = dept.departamento WHERE e.activo = 1 AND e.salario > dept.promedio_dpto ORDER BY e.salario DESC;

Lo que aprendiste

  • Una subconsulta es un SELECT dentro de otro SELECT, siempre entre paréntesis
  • La subconsulta se ejecuta primero y su resultado alimenta a la consulta exterior
  • En WHERE con = usa subconsulta escalar (un valor); con IN usa subconsulta de lista
  • IN es simple pero problemático con NULLs; EXISTS es más seguro y eficiente
  • En FROM crea una tabla derivada temporal — siempre necesita un alias AS
  • En SELECT calcula un valor por cada fila (subconsulta correlacionada)
  • En HAVING compara estadísticas de grupos contra valores calculados dinámicamente
  • NOT IN con NULLs no devuelve filas — preferí NOT EXISTS en esos casos
  • JOIN es generalmente más rápido; subconsulta es más clara para lógica compleja
